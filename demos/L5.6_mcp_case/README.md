# Урок 5.6 — Кейс: Claw-like агент + MCP-витрина данных

**Задача кейса:** бизнес-пользователь ставит цель словами («исследуй, в каком
сегменте мы теряем выручку, и предложи, что проверить»), а автономный Claw-like
агент сам ведёт исследование, обращаясь к корпоративным данным **через MCP** —
не напрямую к БД, а через стандартный протокол с описанными инструментами и
правами.

## Почему через MCP, а не напрямую

MCP (Model Context Protocol) — стандартный «разъём» между агентом и источниками
данных. Витрина данных публикует ровно те инструменты, которые разрешены
(`describe_schema`, `run_sql` read-only), а агент их вызывает. Плюсы для бизнеса:

- **единый контур доступа** — один MCP-сервер обслуживает любых агентов;
- **права и аудит на стороне витрины** — агент физически не может выйти за
  разрешённые инструменты (см. урок 5.5 «безопасное встраивание»);
- **переносимость** — сменили агента/модель, MCP-сервер не трогаем.

```
Бизнес-цель ─▶ Claw-like агент ──MCP──▶ MCP-витрина данных ─▶ PostgreSQL
  (словами)     план→исполнение→         describe_schema       (read-only)
                проверка→итерация         run_sql (SELECT)
```

## Как запустить

MCP-сервер витрины уже собран в Модуле 4 — переиспользуем его:

```bash
# терминал 1 — поднять MCP-витрину (read-only SQL к учебной БД)
cd ../L4.1_tool_calling && ./mcp_server.sh        # streamable-http на :7800

# терминал 2 — запустить Claw-like агента поверх MCP
export AI_PROVIDER_TOKEN=...      # ключ LLM-роутера (LMS)
export MCP_URL=http://localhost:7800/mcp
python3 claw_over_mcp.py "Найди сегмент с просадкой выручки и предложи гипотезу"
```

Без поднятого MCP-сервера и учебной БД агент честно сообщит, что витрина
недоступна — сам цикл «план → MCP-вызовы → проверка» от этого не меняется.

## Связь с практикой модуля

Тот же Claw-цикл, что в `../L5.4_claw_agent/`, но инструменты приходят не из
локального кода, а из MCP-витрины. Это и есть «корпоративный контур»: агент
автономен, но действует только через разрешённый разъём к данным.
